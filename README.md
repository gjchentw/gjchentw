## Hi there 👋

---

我有時候會想，程式設計到底帶給我什麼。不是功能、不是版本號，而是那種靜靜坐在螢幕前、試圖讓混沌的想法變成某個有秩序的東西的感覺。這幾個月下來，我好像一直在做這件事，但每次做的方向都不太一樣。

十月的時候，我心裡有一股衝動，想把一個 Go 框架整理得更清楚。不是說它壞掉了，而是它「可以更好」——我對這種感覺很沒有抵抗力。我把測試補起來，把文件寫得更完整，覺得自己像是在替一棟房子重新整理了一遍，然後滿意地關上門。那種小小的完成感，有時候比起完成什麼驚天動地的功能更能讓我喘口氣。

十一月，我陷入了一個更野心勃勃的計畫。我想做一個能讓瀏覽器的本地檔案自動跟雲端同步的函式庫。OPFS、Web Worker、Google Drive——我一頭栽進了瀏覽器底層機制的世界，試圖讓那些非同步的齒輪彼此咬合得剛剛好。有時候改到一半，我會突然覺得自己像在拼一個邊緣已經磨損的拼圖，不確定缺的那塊究竟是邏輯上的漏洞還是自己的理解不夠深。但最後，它跑起來了，我把它發布出去，心裡有一種奇怪的釋放。

十二月，某個深夜，我想到了三角形內角和定理。不是因為有人叫我去證明它，而是我突然很想知道，能不能用 Lean 4 讓自己「看見」那個證明——不只是記住它，而是真的用邏輯把它砌起來，一塊一塊的。那段時間我過得很慢，但也很紮實，每一個步驟都必須說得清楚，沒有辦法靠模糊的直覺過關。我很喜歡這種被迫嚴謹的感覺，像是在一個沒有謊言可以存活的地方工作。

進入新的一年，我開始思考「怎麼做事」這件事本身。我把自己習慣的開發流程整理成一套工具和投影片，試著說清楚我認為的好流程應該長什麼樣子。這是一種奇怪的事情——把自己的習慣攤開來審視，想著如果要說服別人，我要怎麼解釋自己為什麼這樣做、而不是那樣做。做完之後，我並沒有感到多少自信，反而更清楚看見自己還不確定的那些地方。也許誠實本來就比確信更難。

二月，我忽然想幫朋友的 Minecraft 伺服器做一個小插件。聽起來很輕鬆，但我把它做成了一個認真的專案——有投票機制、有多語言支援、有非同步緩衝、有單元測試。我覺得這就是我的問題，或者說，是我的個性：不管做什麼，都會忍不住想把它做「完整」。過程中，我好幾次問自己，這樣的完整是為了誰？是為了用它的玩家，還是為了我自己心裡那個想要把事情做好的聲音？兩個答案都是，我想。

三月，我又回到 Lean，這次想做的是一個互動式自然數遊戲，讓人可以透過一關一關的關卡去學習怎麼證明加法和後繼函數的性質。我在寫 `2 + 2 = 4` 的正式證明時，忍不住笑了出來。這件事本身很荒謬——一個大人在電腦前認認真真地證明二加二等於四——但那個笑，不是嘲笑，而是某種接近喜悅的東西。在形式系統裡，沒有什麼可以被想當然，你必須把每一個最微小的步驟都說清楚，而當你真的說清楚了，那種感覺比任何「功能上線」都要踏實。

我想，這大概就是我這幾個月一直在追的感覺：不是做出什麼有用的東西，而是那種把某件事想清楚、做清楚的片刻。技術只是載體，真正讓我繼續坐在這裡的，是那個片刻。

### 📝 可重複使用的 Prompt

<details>
<summary>點擊展開 — 用於產生技術能力 Square Pie Chart 的 Prompt</summary>

```
根據 GitHub 使用者 {USERNAME} 過去 6 個月的公開 GitHub 活動（repositories、commits、使用的程式語言與框架），分析其技術能力組成，並製作一張 SVG 格式的 Square Pie Chart（方形比例圖 / Treemap）。

要求：
1. 查看該使用者過去 6 個月內有活動的所有公開 repositories
2. 分析每個 repo 的主要語言、框架、工具（例如：Go、JavaScript、Docker、Lean 4 等）
3. 根據 commit 數量、專案複雜度與技術深度，計算每項技術的加權占比（百分比）
4. 生成一張精美的 SVG treemap 圖表，其中：
   - 每個方塊的面積與該技術的占比成正比
   - 使用不同顏色區分各項技術
   - 在方塊內標示技術名稱、百分比、相關專案名
   - 包含標題與時間範圍
5. 在 README 中以表格形式詳細列出每項技術的占比、相關專案與說明
6. 將 SVG 檔案放在 repo 根目錄，並在 README 中引用顯示

範例輸出格式：
- SVG 檔名：tech-skills-treemap.svg
- 圖表類型：Squarified Treemap（方形比例圖）
- 參考風格：Our World in Data 的 CO₂ 排放 treemap
```
